de producción. Por eso los dos puntos de abajo son CRÍTICOS.

---

## 🔴 Dos reglas críticas antes de tocar nada

1. **NO cambies el `APP_KEY`.** Al recrear el `.env`, usa el **mismo `APP_KEY`**
   de las notas privadas. Si pones otro, los datos financieros cifrados quedarán
   **ilegibles para siempre**. (Guárdalo también junto a los backups.)
2. **Saca un backup de la BD ANTES de correr `/_setup`.** La migración de cifrado
   transforma datos en sitio. Toma un export por **phpMyAdmin** (Exportar →
   Personalizado → SQL + gzip) y guárdalo. Es tu punto de retorno.

## Pasos

### 1. Local (ya hecho)
- [x] Todo commiteado y en `main` (`vendor/` y `public/build` van versionados → sin composer/npm en el server).
- [ ] Poner el repo **público** en GitHub (para poder clonar; luego se vuelve privado).

### 2. Backup de seguridad de prod
- [ ] phpMyAdmin → base `oipfutlf_rrhh` → **Exportar** (SQL + gzip) → descargar y guardar.

### 3. Re-clonar (docs/09 §"Actualizar")
- [ ] Git Version Control → 🗑️ borrar repo `recursoshumanos`.
- [ ] File Manager → borrar `repositories/recursoshumanos` si quedó.
- [ ] Create Repository → clonar `https://github.com/PercyTechX/recursoshumanos.git`.

### 4. Recrear el `.env`
- [ ] Copiar la plantilla de **docs/09** con los valores privados de las notas.
- [ ] **Mismo `APP_KEY`** (regla crítica #1).
- [ ] `APP_SETUP_TOKEN=` con un token nuevo (para abrir `/_setup`).
- [ ] **Incluir las 3 variables NUEVAS de backups:**
  ```dotenv
  GRAPH_DRIVE_BACKUPS=IT
  GRAPH_FOLDER_BACKUPS=BACKUP_SISTEMAS/RRHH_Sistemas
  BACKUP_RETENCION_DIAS=30
  ```

### 5. Permisos y handler PHP (al re-clonar SIEMPRE, docs/09 §⚑)
- [ ] `repositories/recursoshumanos` → **755**; `public` → 755; `.htaccess`/`index.php` → 644.
- [ ] Agregar el bloque **`___lsphp`** al inicio de `public/.htaccess` (handler php82).

### 6. Correr la instalación
- [ ] Abrir `https://rrhh.gds.pe/_setup/EL_TOKEN`. Ahora hace, en una visita:
  migraciones + seeders (incl. tipos de cese) + symlink de storage + `graph:ping`
  + **smoke test de backup** (sube un `.sql.gz` a `IT/BACKUP_SISTEMAS/RRHH_Sistemas`).
- [ ] Verificar en la salida: migraciones OK, `graph:ping` verde, `backup OK ✔`.
- [ ] **Vaciar `APP_SETUP_TOKEN`** en el `.env` (deja `/_setup` en 404).

### 7. Post-deploy manual (nuevo por los backups)
- [ ] **Cron Job** (cPanel → Cron Jobs), diario 02:00:
  ```
  0 2 * * *  /usr/local/bin/ea-php82 /home/oipfutlf/repositories/recursoshumanos/artisan backup:crear >> /home/oipfutlf/backup-rrhh.log 2>&1
  ```
  *(confirmar la ruta real del binario PHP 8.2 en yachay).*
- [ ] En SharePoint, confirmar que **`IT/BACKUP_SISTEMAS`** sea de **acceso
  restringido** (contiene PII y datos cifrados).

### 8. Verificación (humo en prod)
- [ ] Login admin; abrir el **Tablero** (se ven KPIs + panel de cumpleaños).
- [ ] Ficha de un empleado con cuenta/sueldo → se **ve normal** (cifrado transparente).
- [ ] Expediente → Subir documento → tipo "Certificado de Trabajo" (nota de fechas aparece).
- [ ] Generar un **Certificado de Trabajo** (PDF) de un cesado.
- [ ] En SharePoint `IT/BACKUP_SISTEMAS/RRHH_Sistemas` está el `.sql.gz` del smoke test.

## Rollback (si algo sale mal)
1. Restaurar el `.sql.gz` del paso 2 en phpMyAdmin (Importar).
2. Confirmar que el `.env` tenga el `APP_KEY` correcto.
3. (El código anterior sigue en GitHub por si hay que volver a un commit previo.)
